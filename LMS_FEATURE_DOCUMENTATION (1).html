0. System-Wide Notes (read this first)

  • Backend context path: server.servlet.context-path = /api/v1 (application.yml). Every @RequestMapping in a controller is therefore prefixed with /api/v1 at runtime. E.g. CategoryAdminController declares /categories → real URL is /api/v1/categories.
  • Frontend base URL: apiClient.ts sets baseURL = http://localhost:8080/api/v1, so frontend service calls like apiClient.get("/categories") correctly resolve to /api/v1/categories.
  • Two parallel Category controllers exist: CategoryAdminController (/categories) is the one actually consumed by the frontend (categoryService.ts). CategoryController (/api/categories, with @Valid validation) is not called by any frontend code — it produces dead URL /api/v1/api/categories and appears to be leftover/duplicate code from an earlier iteration. Both ultimately hit the same categories table via different service stacks (CategoryAdminService vs CategoryService).
  • FileUploadController (/api/v1/files/upload, local disk storage) is also not used by the frontend. Media uploads in the wizard instead go directly from the browser to Cloudinary (see Feature 5: Content Block Management). This controller is dead code / an earlier prototype.
  • No authentication/authorization is implemented. apiClient.ts injects a Bearer token from localStorage.getItem("token") if present, but nothing in the app ever writes that key, and the backend has no Spring Security dependency, no JWT filter, and no @PreAuthorize checks. All endpoints are open. The Header/Sidebar "Sign Out", "Security", "Developer Console" actions are UI-only alert() simulations.
  • Soft deletes: Categories, Courses, and Content blocks are never hard-deleted; DELETE endpoints flip is_active to false.
  • App is a single-page, state-machine SPA — there is no React Router. Navigation is a currentView string ("courses" | "categories" | "wizard" | "course-detail") held in AppContext and persisted to localStorage.

Feature 1: Category Management

1. Feature Name

Category Management (Course Taxonomy / Curriculum Categories)

2. Feature Purpose

Lets an LMS administrator create, list, edit, and soft-delete the categories that courses are grouped under (e.g. "Cloud Architecture", "Cybersecurity", "Software Engineering"). Categories carry a display icon, brand color, and description, and are used throughout the catalog UI for filter chips and color-coded badges. There is no end-user/learner role in this codebase — every screen is effectively an admin console.

3. Feature Location

Frontend - Page (wired/active): frontend/src/pages/CategoryManagement.tsx - Service: frontend/src/services/categoryService.ts - Types: frontend/src/types.ts (Category interface) - ⚠️ Dead/unused alternative UI: frontend/src/components/CategoryManagementView.tsx (reads/writes via AppContext, never imported by App.tsx) - ⚠️ Dead/unused alternative form: frontend/src/components/CreateCategoryForm_NextJS.tsx

Backend - Controller (active): backend/.../category/controller/CategoryAdminController.java → /categories - Controller (dead): backend/.../category/controller/CategoryController.java → /api/categories - Service interface/impl (active): CategoryAdminService.java / CategoryAdminServiceImpl.java - Service interface/impl (dead, Redis-cached): CategoryService.java / CategoryServiceImpl.java - Repository: CategoryRepository.java - Entity: entity/learning/CategoryEntity.java - DTOs: CategoryRequestDTO.java, CategoryResponseDTO.java - Mapper: CategoryMapper.java (MapStruct, only used by the dead CategoryServiceImpl)

4. Related Files

7. API Details

Method Endpoint (effective, incl. /api/v1) Request Body Response Auth Notes
GET /api/v1/categories — CategoryResponseDTO[] None Returns all categories (active + inactive), each with courseCount
GET /api/v1/categories/{id} — CategoryResponseDTO None 404 if not found
POST /api/v1/categories CategoryRequestDTO (name, icon, description, color, isActive) CategoryResponseDTO (201) None courseCount forced to 0 on create
PUT /api/v1/categories/{id} CategoryRequestDTO CategoryResponseDTO None Recomputes courseCount via courseRepository.countByCategoryId
DELETE /api/v1/categories/{id} — 204 No Content None Soft delete — sets isActive=false, row remains

Error responses: 404 Not Found (category missing — EntityNotFoundException → mapped by GlobalExceptionHandler), 500 Internal Server Error (generic fallback).

8. Database Flow

Table: categories (id, name UNIQUE, icon, description, color, is_active, created_at, updated_at). Relationship: one-to-many with courses (courses.category_id FK, nullable, no cascade — deleting a category does not delete/cascade its courses; courseCount is computed on the fly with a correlated subquery SELECT COUNT(co) FROM CourseEntity co WHERE co.category = c). - Insert: CategoryAdminServiceImpl.create() → categoryRepository.save(). - Read (list): findAllWithCourseCount() — custom JPQL returning Object[] pairs of (CategoryEntity, Long count). - Read (active only): findByIsActiveTrue() (unused directly by admin path) / findByIsActiveTrueWithCourseCount() (used by the dead CategoryServiceImpl, cached in Redis). - Update: field-by-field mutation on the managed entity, then save(). - Delete: soft delete only (is_active=false); no FK cascade is configured.

9. State Management

  • Server state: TanStack Query useQuery(["categories"], categoryService.getAll) — single source of truth for the category table on this page.
  • Mutations: createMutation, updateMutation, deleteMutation via useMutation, each calling queryClient.invalidateQueries({queryKey:["categories"]}) on success — no optimistic updates, the page always shows server-confirmed state.
  • Local UI state: useState for search query, modal open/close, editing target, and form fields.
  • Separately, AppContext.tsx keeps its own copy of categories in React state + localStorage (educorp_categories) for use by the Sidebar/Catalog/Wizard — this is not synced with TanStack Query's cache; the two state stores can drift until both fetch independently.

10. Component Hierarchy

CategoryManagement (page, default export)
   └─ Modal (internal, local component)
        └─ <form> (name, description, icon picker, color picker, active toggle)
   └─ <table> rows (Edit / Delete actions)

   → CategoryAdminController → CategoryAdminServiceImpl → CategoryRepository
   → PostgreSQL → CategoryRepository → CategoryAdminServiceImpl

12. Dependencies

@tanstack/react-query, axios, lucide-react icons. Backend: Spring Web, Spring Data JPA, Lombok, jakarta.persistence.EntityNotFoundException.

13. Validation Rules

  • Frontend: name required (required attribute + formData.name.trim() guard before submit). No other field is mandatory.
  • Backend (active CategoryAdminController): no @Valid/@RequestBody validation annotations — any payload shape is accepted; name UNIQUE constraint at the DB level will throw on duplicate names (surfaces as a generic 500 today, since no DataIntegrityViolationException handler exists).
  • Backend (dead CategoryController): uses @Valid, but the DTO itself (CategoryRequestDTO) has no Bean Validation annotations (@NotBlank, etc.), so @Valid is effectively a no-op here too.

14. Error Handling

  • Frontend: isError state from useQuery renders a "Server Communication Failed" panel with a Retry button; mutation errors render a small red alert banner in the modal ("Failed to submit form...").
  • Backend: GlobalExceptionHandler converts EntityNotFoundException/ResourceNotFoundException → 404, IllegalArgumentException → 400, MethodArgumentNotValidException → 400 with joined field error messages, anything else → 500 generic message.

15. Security

None implemented (see System-Wide Notes). CORS is open via @CrossOrigin(origins = "*") on the controller, in addition to the global WebConfig CORS mapping which restricts /api/** to localhost:5173/localhost:3000 — the controller-level annotation effectively overrides/widens this for this specific endpoint.

16. Important Classes and Methods

  • CategoryAdminController.create/getAll/getById/update/delete — REST entry points.
  • CategoryAdminServiceImpl.create() — builds entity, defaults isActive.
  • CategoryAdminServiceImpl.softDelete() — flips isActive=false.
  • CategoryRepository.findAllWithCourseCount() / findByIdWithCourseCount() — JPQL with correlated course-count subquery.
  • categoryService.getAll/create/update/delete (frontend) — thin Axios wrappers.

17. Feature Dependencies

Course Management (course count rollup, category dropdown in course wizard), no auth dependency.

18. Possible Improvements

  • Remove dead CategoryController/CategoryService/CategoryMapper or actually route the frontend to the validated, cached path.
  • Add @NotBlank/@Size Bean Validation to CategoryRequestDTO and enable @Valid on the active admin controller.
  • Add a DataIntegrityViolationException handler to return a clean 409 on duplicate category names.
  • Replace the two divergent in-memory state stores (AppContext vs TanStack Query) with a single source of truth.
  • Add pagination/server-side search once the category list grows.

19. Summary

  • Admin-only CRUD over categories, no auth enforced.
  • Active code path: CategoryManagement.tsx → categoryService → CategoryAdminController → CategoryAdminServiceImpl → CategoryRepository.
  • A second, unused, validated + Redis-cached implementation (CategoryController/CategoryService) exists as dead code.
  • Soft delete only; courseCount is computed live via a correlated subquery.
  • TanStack Query drives all server-state caching/invalidation on this page.
  • No Bean Validation actually enforced server-side today.

8. Database Flow

Table courses (id, title, slug UNIQUE, thumbnail_url, short_description, description, level, duration, is_active, category_id FK, created_at, updated_at). One-to-many to modules with cascade = ALL, orphanRemoval = true — deleting a CourseEntity via JPA would cascade-delete its modules/submodules/content, but the admin path never hard-deletes (only soft-delete), so this cascade is currently dormant. findCourseDetailBySlug() uses LEFT JOIN FETCH to eagerly pull modules and submodules in one query (avoids N+1) — but its consuming service method is unimplemented (see above).

9. State Management

  • No TanStack Query is used on this specific view (unlike Category Management) — manual useState/useEffect fetching instead.

  • A separate, paginated, active-only /catalog/courses endpoint exists but is unused by the current UI.
  • Course deletion is soft (flag flip), not cascading.
  • Manual useState/useEffect data fetching (no TanStack Query) on this screen.
  • Category data is joined client-side for color/icon badges.

Feature 3: Course Authoring Wizard — Step 1: Basic Details

1. Feature Name

Course Wizard — Basic Details (Step 1 of 4)

2. Feature Purpose

First step of the multi-step course authoring flow. Captures the course's identity: title, auto-generated/editable slug, parent category, short/long description, thumbnail, difficulty level, and duration. Also auto-saves a draft course row to the database as soon as the admin proceeds, so subsequent steps (curriculum, content) have a real courseId to attach child records to.

3. Feature Location

Frontend - Wizard shell: frontend/src/components/CourseWizard/CourseWizardParent.tsx - Step component: frontend/src/components/CourseWizard/WizardStep1.tsx - Services used: courseService.ts, categoryService.ts

Backend (re-uses Course Management's endpoints) - CourseAdminController.java (POST/PUT /admin/courses) - CategoryAdminController.java (GET /categories for the dropdown)

4. Related Files

Frontend: ✓ CourseWizardParent.tsx, ✓ WizardStep1.tsx, ✓ services/courseService.ts, ✓ services/categoryService.ts, ✓ context/AppContext.tsx (wizardCourseId, wizardStep, showToast) Backend: ✓ CourseAdminController.java, ✓ CourseAdminServiceImpl.java, ✓ CourseRequestDTO.java, ✓ CourseEntity.java, ✓ CategoryRepository.java

5. Feature Entry Point

Sidebar "Create Course" button → startNewCourseWizard() (AppContext)
   sets wizardCourseId=null, wizardStep=1, currentView="wizard"
        ↓
CourseWizardParent renders WizardStep1 with empty courseDraft
        ↓
Admin fills Title/Category/Description/Thumbnail/Level/Duration
        ↓
Clicks "Continue" → handleNextStep()

6. Complete Feature Lifecycle

Admin fills WizardStep1 form fields (local useState `localForm`, synced up via setCourseData)
        ↓
Clicks "Continue" → CourseWizardParent.handleNextStep()
        ↓
Step-1 client-side guard: title required, categoryId required (else showToast error, blocks advance)
        ↓
handleSaveDraftSilent() builds CourseRequestDTO-shaped payload (auto-slugifies title if no slug set)
        ↓
IF wizardCourseId is null  → courseService.create() → POST /admin/courses
ELSE                       → courseService.update() → PUT  /admin/courses/{id}
        ↓
CourseAdminController.create()/update() → CourseAdminServiceImpl → CourseRepository.save()
        ↓
Response returns the real DB-generated course id
        ↓
setWizardCourseId(newCourse.id) — wizard now "knows" the real course row
        ↓
setWizardStep(2) → WizardStep2 (Curriculum) renders, using the new real courseId

7. API Details

Re-uses Course Management's POST/PUT /api/v1/admin/courses (see Feature 2 §7). Category options come from GET /api/v1/categories (TanStack Query, categoryService.getAll).

8. Database Flow

Same courses table as Feature 2. This step is the only place a brand-new courses row is actually inserted; subsequent wizard steps always operate against the now-known courseId.

9. State Management

  • WizardStep1 keeps a local localForm (useState) seeded from the parent's courseDraft, with its own isManualSlug flag (auto slug-from-title generation stops once the admin manually edits the slug field).
  • The parent CourseWizardParent owns the canonical courseDraft object (plain useState<any>, not strongly typed) which is threaded down as props (courseData/setCourseData) to all four steps — a single shared draft object pattern rather than per-step isolated state.
  • wizardCourseId and wizardStep are persisted to localStorage (educorp_wizard_id, educorp_wizard_step) via AppContext, so a page refresh resumes the wizard where it left off.

10. Component Hierarchy

CourseWizardParent
 └─ WizardStep1 (courseData, setCourseData)
      ├─ Title/Slug inputs
      ├─ Category select (from categoryService.getAll via useQuery)
      ├─ Description fields
      ├─ Thumbnail input/drag-drop area
      └─ Level/Duration selectors

11. Sequence Diagram (Text)

Admin → WizardStep1 (local form) → CourseWizardParent.handleSaveDraftSilent()
   → courseService.create/update → apiClient → CourseAdminController
   → CourseAdminServiceImpl → CourseRepository → PostgreSQL
   → CourseAdminServiceImpl → CourseAdminController → apiClient
   → CourseWizardParent (setWizardCourseId, setWizardStep) → WizardStep2 renders

12. Dependencies

@tanstack/react-query (category dropdown), lucide-react, courseService, categoryService.

13. Validation Rules

  • Frontend only: title and categoryId required before advancing past Step 1 (showToast error otherwise); no format validation on slug/description.
  • Backend: none (same as Feature 2).

14. Error Handling

  • handleSaveDraftSilent() catches network errors and shows a toast: "Database rejected save. Ensure title and category are set."
  • No retry mechanism; the admin must click "Continue" again.

15. Security

None.

16. Important Classes and Methods

  • CourseWizardParent.handleSaveDraftSilent() — the central "upsert" used by every step transition and by "Save Draft"/"Publish".
  • CourseWizardParent.handleNextStep() — Step-1-specific validation + advance.
  • WizardStep1's slug auto-generation (title.toLowerCase().replace(/[^a-z0-9]+/g, "-")...).

17. Feature Dependencies

Category Management (dropdown data source); feeds Course Curriculum (Step 2), Content (Step 3), Settings/SEO (Step 4).

18. Possible Improvements

  • Type courseDraft with a proper Course-shaped interface instead of any.
  • Add slug uniqueness pre-check (debounced GET) before save to give earlier feedback than a 500 from the DB unique constraint.
  • Add client-side schema validation (e.g. zod) shared across all four wizard steps.

19. Summary

  • Step 1 of 4 in the course authoring wizard.
  • Creates the course row in the DB on first "Continue" (or "Save Draft"), giving downstream steps a real ID.
  • Slug auto-generated from title unless manually overridden.
  • Shares one mutable draft object across all wizard steps via props drilling.
  • Re-uses the same backend endpoints as the main Course Management feature.

7. API Details

Not an HTTP-triggered feature — runs automatically as part of the Spring Boot startup lifecycle (CommandLineRunner).

8. Database Flow

Direct JDBC TRUNCATE ... RESTART IDENTITY CASCADE (raw SQL via JdbcTemplate, bypassing JPA entirely) followed by a sequence of JPA repository.save() calls building the full Category → Course → Module → Submodule → Content hierarchy with real FK relationships.

12. Dependencies

spring-boot-starter-jdbc (via JdbcTemplate), all five JPA repositories, @Transactional.

13. Validation Rules

None — this is trusted internal seed code, not user input.

14. Error Handling

None explicit; any failure here would fail application startup (the whole method is wrapped in @Transactional, so a failure rolls back the partial seed).

15. Security

This is effectively a destructive operation that runs unconditionally on every boot. The class's own doc-comment flags this: "To disable this seeder after the first run, either comment out @Component, wrap in an environment-variable flag, or delete this file." As shipped, every backend restart wipes and re-seeds all categories/courses/modules/submodules/content — a significant risk if ever deployed against real data.

16. Important Classes and Methods

  • DatabaseSeeder.run(String... args) — the entire seeding routine in one method.

17. Feature Dependencies

Touches every entity in the data model; has no feature dependencies of its own.

18. Possible Improvements

  • Gate this behind a Spring profile (e.g. only run when spring.profiles.active=dev) or a config flag — this is the single highest-priority fix in the whole codebase for anyone planning to deploy beyond local development.
  • Replace the raw TRUNCATE with idempotent "seed only if empty" logic.
  • Move seed data to external SQL/JSON fixtures instead of hard-coded Java.

19. Summary

  • Runs automatically on every backend startup via CommandLineRunner.
  • Hard-wipes contents, submodules, modules, courses, categories via raw SQL TRUNCATE ... CASCADE, then re-seeds demo data.
  • Only one course ("Modern Backend Architectures in Java") gets a full demo curriculum with all content block types.
  • Not safe for any environment with real data — must be disabled/gated before production use.

Feature 10: Cross-Cutting Infrastructure

These aren't user-facing "features" but are foundational and touched by every feature above.

10.1 Centralized Axios Client (apiClient.ts)

  • Single Axios instance, baseURL from VITE_API_BASE_URL env var (default http://localhost:8080/api/v1), 15s timeout.
  • Request interceptor: injects Authorization: Bearer <token> if localStorage.getItem("token") exists (nothing in the app currently sets this key — dormant scaffolding for future auth).
  • Response interceptor: classifies errors by HTTP status (401/403/404/5xx) with console.error logging, and normalizes every rejection into a consistent ApiError shape (status, message, details, isNetworkError, originalError) consumed uniformly by every *Service.ts file.

10.2 Global Exception Handling (GlobalExceptionHandler.java)

@RestControllerAdvice mapping ResourceNotFoundException/EntityNotFoundException → 404, IllegalArgumentException → 400, MethodArgumentNotValidException → 400 (joined field messages), and a catch-all Exception → 500 with a generic message — used uniformly across every controller in the app via ErrorResponse (a Java record).

10.3 CORS Configuration

Two overlapping layers: WebConfig.java globally restricts /api/** to localhost:5173/localhost:3000 origins with specific methods/credentials; but every individual controller also carries @CrossOrigin(origins = "*"), which is more permissive and effectively the one that applies per-endpoint. FileUploadController instead hardcodes @CrossOrigin(origins = "http://localhost:3000").

10.4 Redis Caching (RedisConfig.java)

Configures a RedisCacheManager with a 1-hour TTL, JSON value serialization, null-value caching disabled. Only actually exercised by the dead CategoryServiceImpl (@Cacheable("categories")/@CacheEvict) — the live CategoryAdminServiceImpl path used by the frontend does not use caching at all, so Redis is effectively configured but unused in the active request path today.

10.5 MapStruct Mapping

Two @Mapper(componentModel="spring") interfaces (CategoryMapper, CourseMapper) auto-generate entity↔DTO conversion. CategoryMapper is only used by the dead CategoryServiceImpl; CourseMapper is used by the live CourseCatalogServiceImpl/CourseAdminServiceImpl... actually note: CourseAdminServiceImpl does not use CourseMapper either — it has its own manual mapToResponse(). Only CourseCatalogController's read paths benefit from MapStruct today.

10.6 Soft-Delete Convention

Every primary entity (CategoryEntity, CourseEntity, ModuleEntity, SubmoduleEntity, ContentEntity) carries an is_active boolean defaulting to true. Every DELETE endpoint in the app flips this flag rather than removing the row — meaning nothing is ever permanently deleted through the API, only through the DatabaseSeeder's raw TRUNCATE.
